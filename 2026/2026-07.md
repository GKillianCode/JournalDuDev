##### Article : Yagni par Martin Fowler

[Lien de l'article ici](https://martinfowler.com/bliki/Yagni.html)

Yagni pour `You Aren't Gonna Need It` : `Tu n'en auras pas besoin`.

Ce principe de programmation extrême dit : tant que la fonctionnalité n'est pas nécessaire, on ne la code pas, car elle a un impact sur le coût de développement et de mise en place, mais également sur les délais et la maintenabilité, au détriment des autres fonctionnalités.

##### Design-First

En tirant le fil après l'article sur la méthode Yagni, j'ai découvert l'approche Design-First. Cette approche consiste à figer le besoin utilisateur dans un contrat OpenAPI.

Avant même de commencer à coder, le Design-First apporte une rigueur dans la conception de l'application à travers un fichier YAML contenant les différents endpoints, les différentes réponses possibles avec leurs headers et, en option, les schémas de sortie.

En plus de s'inspirer des principes de Yagni, cette approche permet au développeur frontend d'implémenter le frontend en même temps que les développeurs backend implémentent le backend.

#### Lecture
##### Article : Approche Design-First Du Développement D'API : Comment L'implémenter Et Pourquoi Cela Fonctionne

[Lien de l'article ici](https://www.infoq.com/fr/articles/design-first-api-development/)

**Le Code-First :** Cette approche permet une progression rapide du développement d'une application, mais au détriment de la complexité de l'API et des personnes non développeuses qui l'utiliseront.
    
**L'API-First :** Cette approche considère l'API comme l'élément central de l'application et consiste à rédiger un contrat dans un langage tel qu'OpenAPI (YAML). L'inconvénient est que le langage de rédaction du contrat peut devenir limitant, voire bloquant.
    
**Le Design-First :** Cette approche considère que l'API est l'élément commun entre toutes les parties prenantes. Elle doit donc être définie et validée de manière itérative par tous.

#### Design-First

Dans l'article précédent, Code-First, API-First et Design-First sont tous les trois au même niveau, alors qu'en réalité, il existe principalement deux approches : le Code-First et l'API-First.

Le Design-First, quant à lui, n'est qu'une manière d'appliquer l'API-First.

En creusant le sujet, et notamment la manière de l'appliquer, j'ai découvert que le contrat peut être écrit en YAML (le langage majoritairement utilisé) ou en JSON.

Avec YAML (celui sur lequel j'ai fait mes recherches), il est possible, et même fortement conseillé, de découper le contrat en plusieurs fichiers. Il est également possible de créer des composants réutilisables.

Finalement, le Design-First apporte une véritable consolidation de l'API en donnant un cap.

- fixe dans le contrat la manière dont les endpoints sont organisés et hiérarchisés ;
- fixe toutes les sorties possibles ainsi que les codes d'erreur ;
- permet le versionnement de l'API ;
- et surtout, permet aux développeurs frontend de développer le frontend pendant que les développeurs backend développent le backend...

#### RFC-2119

[Lien de la RFC-2119]([https://www.infoq.com/fr/articles/design-first-api-development/](https://www.ietf.org/rfc/rfc2119.txt))

Cette spécification préconise l'utilisation de certains mots-clés, qui ont chacun leur poids, comme : "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY" et "OPTIONAL".

OpenAPI, dans le cadre du Design-First, utilise `required` pour un champ obligatoire, rien de spécifié pour un champ optionnel et `deprecated` pour recommander de ne plus utiliser un champ.

#### Java

Faits intéressants sur les valeurs par défaut.

```java
int number; // type primitif : valeur par défaut = 0
float floatnumber; // type primitif : valeur par défaut = 0.0

char letter; // type primitif : valeur par défaut = \u0000 (caractère nul)

String myStringSentence; // type objet : valeur par défaut = null
String myStringSentence2 = null; // type objet : valeur par défaut = null
```

Si l'on essaie d'imprimer dans la console `myStringSentence` et `myStringSentence2`, les résultats sont différents :
- `myStringSentence` retourne une `NullPointerException` si celle-ci n'est pas une propriété de classe.
- `myStringSentence2` affiche `"null"` dans la console sous forme de chaîne de caractères.

Java ajoute des valeurs par défaut uniquement aux propriétés de classe. Dans les autres cas, Java retourne une erreur de type `Variable might not have been initialized`.

#### Lecture

##### Article : Richardson Maturity Model

[Lien de l'article de Martin Fowler](https://martinfowler.com/articles/richardsonMaturityModel.html)
[Lien de l'article de Devopedia](https://devopedia.org/richardson-maturity-model)
[Lien de l'article Wikipedia](https://fr.wikipedia.org/wiki/Mod%C3%A8le_de_maturit%C3%A9_de_Richardson)

Le Modèle de Maturité de Richardson est un modèle qui permet de classer les API en quatre niveaux distincts.

**Niveau 0 :** Le niveau de modèle d'API le plus simple qui existe.

**Niveau 1 :** Ce niveau considère chaque donnée comme une ressource avec sa propre URI.

**Niveau 2 :** Ce niveau introduit les verbes HTTP (_safe operations_), les codes de réponse et les en-têtes de réponse.

**Niveau 3 :** Ce dernier niveau ajoute des contrôles dits hypermédia. Il permet notamment d'ajouter des liens hypertexte dans les réponses afin que le client puisse découvrir les actions possibles. Ce niveau porte le nom de **HATEOAS** (_Hypertext As The Engine Of Application State_).

De nos jours, le niveau 2 est le plus pratiqué en entreprise. Le niveau 3 ajoute une grande complexité de mise en place.

##### Strings

```java
String mySimpleString = "Hello world";
String myObjectString = new String("Hello World");
```

Les deux façons d'initialiser des `String` sont techniquement totalement différentes :
- pour l'initialisation de `mySimpleString`, la JVM utilise un système appelé **Java String Pool**. Ce système optimise la gestion de la mémoire des `String` et fait référencer plusieurs contenus identiques vers la même référence mémoire.
- l'initialisation de `myObjectString` force la JVM à créer un nouvel objet, même si un objet identique existe déjà en mémoire.

#### Lecture
##### Article : Guide to Java String Pool - baeldung.com
[Lien de l'article](https://www.baeldung.com/java-string-pool)

Le **String Pool** est un espace mémoire spécifique dans lequel la JVM stocke les `String`.

La JVM optimise sa mémoire en ne conservant qu'une seule copie de chaque littéral dans sa mémoire.

Avant Java 7, la JVM plaçait les `String` (créées sans `new`) dans un espace nommé **PermGen**. L'inconvénient était que cet espace était fixe et non extensible, ce qui pouvait provoquer des erreurs de type **OutOfMemoryError**.

Jusqu'à Java 8, les `String` étaient représentées en mémoire par un tableau de caractères encodés en UTF-16.

À partir de Java 9, les `String` sont représentées en mémoire sous forme de **Compact Strings**. Ce nouveau format choisit automatiquement le mode de stockage le plus adapté entre un tableau de caractères UTF-16 et un tableau de bytes encodés en Latin-1.

Depuis Java 9, les `String` ne sont donc encodées en UTF-16 que lorsque cela est nécessaire. Grâce à cela, l'empreinte mémoire a été significativement réduite.

#### Lecture
##### Article : Autoboxing and Unboxing in Java
[Lien de l'article](https://www.geeksforgeeks.org/java/autoboxing-unboxing-java/)

**L'autoboxing** transforme les types primitifs en objets, ce qui permet notamment d'utiliser les classes _wrapper_ et, dans certains cas, de réutiliser des références présentes dans le cache.

À l'inverse, **l'unboxing** transforme les objets en types primitifs.

Par exemple, lors de la création d'une liste d'entiers en Java, nous utilisons le _wrapper_ `Integer` (pour le type primitif `int`). Lorsqu'un _wrapper_ est utilisé dans le code, l'autoboxing s'applique automatiquement dans certaines situations.

Les objets `Integer` sont mis en cache pour les valeurs comprises entre **-128** et **127**, soit l'intervalle des valeurs représentables par un `byte`. Cela signifie que deux autoboxings d'une même valeur comprise dans cette plage référenceront le même objet. En revanche, au-delà de cette plage, une nouvelle instance est généralement créée.

Ce mécanisme fonctionne de manière similaire pour certains autres types primitifs et leurs _wrappers_, notamment `Byte`, `Short`, `Long`, `Character` et `Boolean`.

Les `Float` et les `Double`, en revanche, ne bénéficient pas de ce mécanisme de cache.

Enfin, les `String` sont stockées dans la **Java String Pool**. Les chaînes identiques partagent la même référence mémoire, à condition qu'elles ne soient pas créées avec le mot-clé `new`.

##### Text Blocks

Les _Text Blocks_ rendent l'écriture de texte sur plusieurs lignes plus lisible pour les développeurs et permettent de ne plus utiliser `\n` ou `\t`, notamment pour générer du code HTML ou JSON.

Exemple :
```java
String html = """ 
	<html> 
		<body> 
			<p>Hello world</p> 
		</body> 
	</html> 
""";
```

##### Var
`var` peut être considéré comme un raccourci visant à améliorer la lisibilité du code. Il ne peut être utilisé que pour les variables locales dont l'initialisation permet au compilateur de déduire explicitement le type.

À la compilation, le compilateur remplace `var` par le type explicite correspondant.

Par exemple, dans le cas d'une liste contenant à la fois des `String` et des `Integer`, le compilateur ne plantera pas. Il déduira le type commun le plus générique, à savoir `Object`.

Le seul inconvénient, dans cet exemple, est que les éléments sont considérés comme des `Object`. Les méthodes spécifiques aux types `String` ou `Integer` ne sont donc plus directement accessibles. La seule solution consiste à caster les objets vers leur type réel lorsque cela est nécessaire.

#### Lecture
##### Article : Raw Types in Java
[Lien de l'article](https://www.baeldung.com/raw-types-java)

Les _Raw Types_ (ou **types bruts** en français) correspondent à l'utilisation d'une interface ou d'une classe générique sans préciser son type d'argument.

Exemple :
```java
List list = new ArrayList(); // raw type
List<Integer> listIntgrs = new ArrayList<>(); // parameterized type
```

Ce type est déconseillé pour trois raisons :
- il n'est pas lisible (expressif) ;
- il ne fait pas partie des types _safe_ ;
- il ne compile parfois pas.

##### Article : JVM Garbage Collectors
[Lien de l'article](https://www.baeldung.com/jvm-garbage-collectors)

On entend souvent parler du **Garbage Collector**, mais en réalité, il n'y en a pas qu'un.

Il existe différents types de Garbage Collector :
- Serial Garbage Collector ;
- Parallel Garbage Collector ;
- G1 Garbage Collector ;
- Z Garbage Collector.

Chaque GC est optimisé selon trois facteurs :
- **Le débit (Throughput)** : maximiser le temps pendant lequel l'application travaille par rapport au temps passé à nettoyer la mémoire.
- **La latence (Pause Time)** : minimiser la durée des pauses pendant lesquelles l'application est complètement arrêtée (les fameuses pauses _Stop-The-World_ ou STW).
- **L'empreinte mémoire (Footprint)** : la quantité de RAM nécessaire pour que le GC fonctionne efficacement.

###### Le Serial Garbage Collector (SGC)
Le plus simple et le plus utilisé dans les anciennes applications. Il fonctionne sur un seul thread.

Lorsqu'il nettoie la mémoire, l'application est complètement à l'arrêt.

###### Le Parallel Garbage Collector (PGC)
Ce GC est une évolution du précédent avec un fonctionnement en multithread, optimisant ainsi le débit et le temps de pause.

###### Le G1 (Garbage First) Garbage Collector (G1GC)
Toujours dans un objectif d'optimisation, ce GC découpe la mémoire en un grand nombre de _chunks_, contrairement aux précédents qui travaillaient sur de gros _chunks_.

Le principe du G1 est différent de celui des précédents, car il surveille les différents _chunks_ et nettoie en priorité ceux contenant le plus d'objets morts.

Il ajoute également un paramètre permettant de définir un temps de pause maximal.

Depuis Java 9, c'est le GC utilisé par défaut.

###### Le Z Garbage Collector (ZGC)
Ce GC est le dernier à être arrivé en production, depuis Java 15.

Il est spécialisé pour les applications à très faible latence. Il est capable d'effectuer la majorité de ses tâches de manière concurrente, ce qui rend les temps de pause presque inexistants.

##### Article : A deep dive into Java garbage collectors
[Lien de l'article](https://www.datadoghq.com/blog/understanding-java-gc/)

Les GC ont été créés pour faciliter la vie des développeurs et éviter les risques suivants :
- **memory leaks** (fuites de mémoire) : oubli de libérer de la mémoire réservée ;
- **dangling pointers** (pointeurs pendants) : pointeurs vers une référence mémoire qui n'existe plus ;
- **double frees** (double libération) : libération d'un espace mémoire déjà libéré.

Dans la mémoire (le tas), la JVM découpe sa stratégie de gestion de la mémoire en deux grandes parties :
- la **Young Generation** ;
- la **Old Generation**.

Avec ce fonctionnement, le Garbage Collector passe beaucoup plus souvent dans la **Young Generation** que dans la **Old Generation**, ce qui lui permet de réduire son périmètre d'action et de se concentrer uniquement sur une petite portion de la mémoire.

Lorsqu'un objet passe plusieurs cycles de nettoyage (l'Eden, le S0 et le S1), il est promu vers la génération suivante. Après avoir traversé l'Eden (l'espace de création des objets) ainsi que les deux espaces de survie (_Survivor Spaces_), il est promu dans la **Old Generation**.

Lors du passage d'un espace à un autre, les objets sont regroupés les uns à côté des autres en mémoire, ce qui présente plusieurs avantages :

- une création d'objets très rapide (_Bump Pointer Allocation_) ;
- une amélioration des performances du processeur grâce à la localité mémoire (_Memory Locality_).

Selon le contexte de l'application, il est très important de choisir le bon GC en fonction de la priorité recherchée : soit la latence (des pauses les plus courtes possible), soit le débit (la vitesse de traitement).

Pour maximiser le débit, le **Parallel GC** est le plus efficace. Pour minimiser la latence, c'est le **ZGC** qui est le plus performant.

Le **G1GC** adopte une approche différente. Au lieu de considérer la **Young Generation** et la **Old Generation** comme deux gros blocs mémoire, il les subdivise en régions (_regions_).

Pour le nettoyage de la **Young Generation**, le G1GC fonctionne comme les autres GC : il supprime les objets inutiles d'un seul coup. Pour la **Old Generation**, c'est là que son fonctionnement diffère. Il observe les différentes régions et nettoie en priorité celles qui contiennent le plus d'objets morts.

##### Article : Data Oriented Programming in Java
[Lien de l'article](https://www.infoq.com/articles/data-oriented-programming-java/)

Définition des paradigmes de programmation :

- <u>La programmation impérative :</u>
    > Tout est une suite d'instructions à exécuter.
    
    Utilisée par : C, Go...
    

<br/>

- <u>La programmation orientée objet (POO) :</u>
    > Tout est un objet.
    
    Utilisée par : Java, C++, C#...
    

<br/>

- <u>La programmation fonctionnelle :</u>
    > Tout est une fonction.
    
    Utilisée par : JavaScript...
    

<br/>

- <u>La programmation déclarative :</u>
    > Tout est une déclaration d'intention.
    
    Utilisée par : SQL, HTML...
    

**La programmation orientée objet (POO) :**

- La philosophie de ce paradigme consiste à dire : je définis des frontières (encapsulation, sécurité, versionnement) et je les défends.
- Idéale, entre autres, pour les grands systèmes monolithiques.

**La programmation orientée data (POD) :**

- La philosophie de ce paradigme consiste à dire : je sépare les données et la logique afin de les rendre universelles et simples.
- Idéale pour les architectures modernes en microservices.

Il n'est pas forcément nécessaire de les opposer. Une architecture logicielle peut tirer parti des deux paradigmes, chacun ayant son domaine d'application.

##### Article : Data Oriented Programming in Java (suite de l'étude)

Dans tous les langages orientés objet, il est possible de travailler en programmation orientée data.

Depuis les versions récentes de Java, il est possible, grâce au projet Amber, de travailler avec le paradigme DOP beaucoup plus facilement grâce à de nouvelles fonctionnalités : l'inférence de type avec les variables locales (`var`), les _Text Blocks_, les _Sealed Classes_ et le _Pattern Matching_, entre autres.

Le mot-clé `sealed` résout la problématique suivante : autoriser l'héritage d'une classe uniquement à un groupe restreint de classes grâce au mot-clé `permits`.

```java
public sealed interface MoyenPaiement permits CarteBancaire, Paypal, VirementInstantane {}

public final record CarteBancaire(String numero) implements MoyenPaiement {} 
public final record Paypal(String email) implements MoyenPaiement {} 
public final record VirementInstantane(String iban) implements MoyenPaiement {}
```


Avec `sealed` vient le _Pattern Matching_. Grâce aux enfants de la classe scellée, le compilateur connaît tous les cas possibles et ne compile pas si tous les cas ne sont pas présents, ce qui représente un énorme gain en sécurité. De plus, puisque tous les cas sont listés, il n'est pas nécessaire de préciser un cas par défaut.

```java
double frais = switch (moyen) {
    case CarteBancaire cb -> 0.02;
    case Paypal p -> 1.50;
    // Le copilateur ne compilera pas car il manque VirementInstantane
};
```


Il est tout à fait possible d'utiliser la POO et le DOP dans un même projet. Une des possibilités consiste à dissocier les classes représentant les acteurs des messages.

Exemple :

- **Acteurs** : Controller, Service, Repository...
- **Messages** : DTO, Payload, Value Object (VO), Configuration...
